POC: Ícone de app dinâmico em React Native

Objetivo

Esta POC tem como objetivo analisar a possibilidade de trocar o ícone do app (launcher) de forma dinâmica.

É possível trocar o ícone de forma dinâmica?

Sim porém as plataformas (iOS e Android) apenas permitem selecionar entre opções que o sistema já conhece a partir do pacote instalado, não trocar o ícone por qualquer arte nova que chegue só da rede.

Para o SO e o launcher, o ícone é identidade do pacote: fica definido na instalação ou numa atualização pela loja.

As APIs oficiais só escolhem variantes já empacotadas no bundle.

Por que as plataformas funcionam assim

  1. Identidade e revisão — Lojas esperam que o “ícone do app” esteja ligado ao binário revisto/assinado; ícones totalmente arbitrários em runtime abriam margem a confusão e abuso (imitação de outras apps, conteúdo impróprio sem passar pelo mesmo escrutínio do pacote).
  2. Contrato explícito — Tanto CFBundleAlternateIcons como activity-alias são declarações: “estas são as entradas e artes permitidas”.

Isto não impede a app de descarregar imagens para uso dentro da UI (splash, avatar, marketing); só não substitui o mecanismo oficial de ícone na gaveta descrito acima.

Como implementar a troca de ícone de forma dinâmica?

Onde colocar os ícones

  • iOS: no Asset Catalog, em geral em Images.xcassets (conjuntos .appiconset por variante).
  • Android: em res/mipmap-* (por exemplo mipmap-hdpi, mipmap-xxhdpi, …) dentro de android/app/src/main/res/.

Bibliotecas de terceiros

Existem opções como:

  • react-native-change-icon
  • @computools/react-native-dynamic-app-icon

Ao analisá-las, foi verificado que muitas estão pouco mantidas ou desatualizadas, o que aumenta o risco em atualizações futuras do React Native.

Implementação nativa como solução

É preciso um módulo nativo para ligar o JavaScript às APIs de cada plataforma.

iOS

  • setAlternateIconName / alternateIconName: só aceita nomes já declarados no Info.plist em CFBundleIcons → CFBundleAlternateIcons, cada um ligado a um set no Asset Catalog do bundle.
  • Não há API para “inventar” novos ícones em runtime por download: sem plist + assets na mesma build, não há candidato na home. Depois da build, só se escolhe entre o que já veio no pacote.

Exemplo de alternativa em Info.plist:

<key>CFBundleAlternateIcons</key>
<dict>
    <key>SolIcon</key>
    <dict>
        <key>CFBundleIconFiles</key>
        <array>
            <string>ic_launcher_sol</string>
        </array>
    </dict>
</dict>

No Xcode, garante que os sets de ícone alternativos entram no target (p.ex. incluir todos os assets de ícone da app, conforme a opção disponível na tua versão).

Android

  • Um activity-alias por variante de ícone, todos a apontar para a MainActivity; PackageManager.setComponentEnabledSetting liga só o alias desejado e getComponentEnabledSetting lê o estado.

Exemplo de alias:

<activity-alias
    android:name=".MainActivitySol"
    android:targetActivity=".MainActivity"
    android:icon="@mipmap/ic_launcher_sol"
    android:roundIcon="@mipmap/ic_launcher_sol_round"
    android:enabled="true"
    android:exported="true" />

Ajusta nomes, mipmaps e o intent LAUNCHER ao teu manifesto; normalmente um só alias fica enabled="true" de cada vez.
